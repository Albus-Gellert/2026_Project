# Global Water Stress Explorer — 高保真可视分析原型

这是浙江大学可视化课程 Project 2 的 UI-first 原型。项目暂时以全球水资源、水压力与行业用水结构为分析主题，使用 Vue 3、TypeScript、Vite 和 Apache ECharts 实现。

当前版本的重点不是给出最终研究结论，而是先建立一套完整、可讨论的分析界面：用户如何限定观察范围、如何从全球分布进入国家个案、如何比较国家位置、如何探索指标关系，以及如何把图表观察收束为后续需要验证的问题。

> **数据声明：** `src/mockData.ts` 中的全部数值都是 AQUASTAT-style 模拟数据，仅用于演示界面、图表和联动。它们不能被引用为 AQUASTAT 事实数据，也不能作为课程项目的最终研究结论。

## 01–06 的整体叙述逻辑

01–06 不是六个独立路由页面，而是同一张长页面中的六段分析叙事。用户按从上到下的阅读顺序，完成下面这条分析路径：

```text
01 界定分析范围
   ↓
范围摘要建立当前基准
   ↓
02 在空间上定位问题 ── 03 解释当前国家画像
   ↓
04 判断相对位置 ────── 05 探索指标关系
   ↓
06 把观察收束为待验证的问题
```

这套结构对应三个递进层级：

1. **Overview：先看整体。** 通过筛选、摘要和地图回答“现在看的是谁、哪里、哪一年、哪个指标”。
2. **Comparison：再做比较。** 通过国家画像、排名和关系视图回答“当前国家处在什么位置，它与其他国家有什么差异”。
3. **Interpretation：最后形成问题。** 发现工作台把前面的视觉证据整理成观察提示，为真实数据分析提出下一步问题，而不是直接宣称因果关系。

## 页面模块说明

### 01 · Explore the system — 界定分析范围

**这一模块在讲什么**

01 是整段分析的入口。用户先选择年份、区域和主指标，也可以搜索国家或重置视图。这里决定后续所有图表所使用的观察范围和共同语境。

**分析逻辑**

- `Year` 决定时间切片。
- `Region` 决定参与比较的国家集合。
- `Map & ranking metric` 保证地图和排名使用同一个主指标，避免两个视图比较口径不一致。
- `Find country` 决定当前重点解释的国家，但不会改变指标定义。
- 下方未编号的 `SummaryStrip` 是 01 与后续图表之间的桥梁：它给出当前范围的平均水压力、总取水量、高压力国家数和覆盖人口，使后续的“高、低、异常”有一个明确基准。

**排版逻辑**

模块横跨整页，控件在宽屏上横向排布，让用户在进入图表前一次看清所有分析条件。范围摘要紧接在筛选器下方，采用等宽数字卡片，把控制动作转换成可读的当前样本概况。

**交互结果**

改变任一筛选项会更新地图、画像、排名、散点和发现工作台。如果区域筛选使当前国家不再可见，系统会自动选择该区域内水压力最高的模拟国家；搜索区域外国家时，区域会自动回到 `All regions`。

### 02 · Spatial Overview — 回答“问题在哪里”

**这一模块在讲什么**

02 用世界地图展示当前主指标在不同国家的空间分布。颜色深浅编码指标值，灰色区域表示当前模拟数据没有覆盖；选中国家使用独立高亮。

**分析逻辑**

地图适合先发现区域聚集、空间差异和需要继续查看的国家。它回答的是“哪里高、哪里低、是否存在成片分布”，而不是解释造成差异的原因。

**排版逻辑**

地图位于 Overview 行左侧，宽度约占该行的三分之二。地理形状需要较大的横向画布，因此它被设置为本行的主视觉；用户的视线先扫过整体分布，再自然转向右侧的国家画像。

**交互结果**

地图支持拖动、缩放、悬停查看数值和点击国家。点击后，03、04、05、06 会共同切换到同一国家。

### 03 · Country Profile — 回答“这个国家具体怎样”

**这一模块在讲什么**

03 对当前选中国家做纵向画像，包括：

- 当前主指标在筛选范围内的名次；
- 水压力风险提示；
- 可再生水资源、总取水量、用水效率和灌溉面积；
- 农业、工业和城市用水构成；
- 水压力、用水效率和人均资源相对于区域平均值的位置。

**分析逻辑**

地图只能告诉用户“在哪里”，国家画像继续回答“它由什么构成、与所属区域相比有何不同”。这里把单个值拆成资源、取用、行业结构和区域基准，避免仅凭地图颜色判断国家状况。

**排版逻辑**

画像位于地图右侧，形成“全球分布 + 个体解释”的主从关系。它使用较窄的纵向卡片和紧凑的小指标、环形图、基准条，适合沿一个国家向下阅读，不与地图争夺视觉中心。

### 04 · Country Ranking — 回答“相对位置有多高”

**这一模块在讲什么**

04 将当前范围内的国家按主指标降序排列，默认展示前 10 名。若当前国家不在前 10，仍会把它追加显示，以保证用户能够看到它在同一量尺上的位置。

**分析逻辑**

地图的颜色适合看空间模式，但不适合精确比较相近数值。排名通过统一基线和明确顺序回答“谁最高、差距多大、当前国家排在哪里”，是对 02 的定量补充。

**排版逻辑**

排名位于第二个分析行左侧，宽度略小于散点图。横向条形图保留了足够的国家名称空间，同时让右侧更宽的二维关系视图获得更多坐标空间。

**交互结果**

点击任意条形会选中相应国家，并同步更新其余模块。

### 05 · Relationship View — 回答“哪些指标可能共同变化”

**这一模块在讲什么**

05 允许用户自由选择 X 轴和 Y 轴指标，观察国家在两个变量之间的分布关系：

- 点的位置表示两个指标的取值；
- 颜色表示区域；
- 气泡大小表示人口；
- 当前国家使用金色描边和名称标签；
- 当数值跨度超过 100 倍时，坐标轴自动使用对数尺度。

**分析逻辑**

散点图用于发现相关方向、区域分群和离群点，为“哪些因素值得继续检验”提供线索。它只表达共同变化和分布模式，不能单独证明因果关系。

**排版逻辑**

散点图位于排名右侧并获得更宽空间，因为二维坐标、区域图例、气泡和轴选择器需要同时保持可读。这样一行内形成“左侧回答顺序，右侧探索关系”的分析分工。

**交互结果**

用户可以分别切换 X/Y 指标，也可以点击任意气泡切换当前国家。指标切换不会改变 02 和 04 共用的主指标，因而“主比较问题”和“辅助关系探索”保持相互独立。

### 06 · Findings Workbench — 回答“下一步值得验证什么”

**这一模块在讲什么**

06 将前面视图中的状态整理为三类演示观察：

- `Current Observation`：当前国家相对筛选范围均值的差异；
- `Country Comparison`：当前国家在所属区域中的排名与区域均值；
- `Possible Outlier`：基于标准差距离的候选离群点提示。

**分析逻辑**

这一模块把“看图”转成可讨论的研究问题，例如：为什么当前国家高于范围均值、区域差距可能由什么结构造成、某个离群点是否来自真实机制或数据质量问题。现有文案由模拟值自动生成，只是叙事模板，不是最终洞察。

**排版逻辑**

06 横跨整页并使用深色背景，作为整条叙事的视觉结尾。三张观察卡并排对应三种分析层次，页脚再明确提示下一步应使用真实 AQUASTAT 数据验证。深色区也把“图表证据”与“解释和假设”区分开，提醒读者不要把自动提示当成事实结论。

## 整体排版原则

宽屏下，页面从上到下分为五层：

| 层级 | 内容 | 宽屏布局 | 设计目的 |
| --- | --- | --- | --- |
| 页首 | 标题、当前国家/年份、Mock Data 标签 | 通栏 | 建立主题与数据状态 |
| 输入层 | 01 筛选器 + 范围摘要 | 通栏 | 先固定比较口径 |
| 总览层 | 02 地图 + 03 国家画像 | 约 `2.05 : 0.95` | 从整体空间进入个体解释 |
| 分析层 | 04 排名 + 05 散点图 | 约 `0.84 : 1.16` | 从精确比较进入关系探索 |
| 收束层 | 06 发现工作台 + 数据声明 | 通栏 | 整理观察并提出下一步问题 |

当视口宽度低于 `1150px` 时，两组双栏都会变为单栏；筛选控件与观察卡也会继续重排。因此移动端仍保持 01 → 06 的叙述顺序，只改变并排关系，不改变分析逻辑。

## 跨模块联动逻辑

六个共享状态集中在 `App.vue`：

- `selectedCountry`
- `selectedYear`
- `selectedRegion`
- `selectedMetric`
- `selectedXMetric`
- `selectedYMetric`

典型联动流程如下：

```text
用户点击地图、排名条形或散点气泡
  → 子组件发送 selectCountry
  → App.vue 更新 selectedCountry
  → 地图高亮、国家画像、排名、散点标签和发现工作台共同刷新
```

`App.vue` 负责共享状态、筛选结果和页面组合；各组件只负责自己的表达方式。指标标题、单位、格式和配色统一从 `metricConfig.ts` 读取，从而避免同一指标在不同模块中出现口径差异。

## 环境要求与运行方式

- Node.js 18 或更高版本；课程材料推荐 Node.js 22 LTS。
- pnpm。

进入可运行项目目录：

```powershell
cd 2025_ZJU-SummerSchool-project-template-main
```

首次安装并启动：

```powershell
pnpm install
pnpm run dev
```

根据终端输出打开本地地址，通常是 `http://localhost:5173/`。

检查 TypeScript 并生成生产构建：

```powershell
pnpm run build
```

## 模拟数据结构

每条 `CountryYearRecord` 包含：

- `country`、`iso3`、`region`、`year`
- `population`：百万人
- `waterStress`：百分比
- `renewableWaterResources`：km³/yr
- `totalWaterWithdrawal`：km³/yr
- `agriculturalWithdrawal`、`industrialWithdrawal`、`municipalWithdrawal`：km³/yr
- `waterUseEfficiency`：本原型中的合成指数
- `irrigatedArea`：百万公顷

模拟数据不是让每个字段独立随机变化：行业用水由总取水量和行业份额推导，年份变化由地区人口增长、资源波动和取水趋势共同产生，因此不同图表之间可以呈现一致关系和少量刻意设置的离群点。

## 后续替换为真实数据

1. 将真实 AQUASTAT 数据预处理为 `types.ts` 中的 `CountryYearRecord`，或按最终变量修改该接口。
2. 保持 `country + year` 唯一，并统一单位；不要在同一字段中混用 m³、百万 m³ 和 km³。
3. 用本地 JSON、CSV 转换结果或 API 加载器替换 `mockData.ts` 的模拟导出。
4. 将国家名称映射到 `public/data/world.json`；名称不一致时应在数据加载阶段维护显式映射表。
5. 在 `metricConfig.ts` 统一更新标题、单位、小数位、说明和颜色。
6. 明确处理缺失值，不要使用 0 代替未知值。
7. 数据验证完成后，删除 `Prototype / Mock Data` 标签，并用有证据支持的分析替换 06 中的演示文案。

更详细的组件关系、数据流和常见修改入口见 [ARCHITECTURE.md](./ARCHITECTURE.md)。
